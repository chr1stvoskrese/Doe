# Аудит безопасности Doe

Дата: 2026-07-11. Проверены: точки входа (`main.py`, `wrapper.py`), IPC-мост
pywebview, все API-роуты (`src/api/v1/*`), крипта хранилища, биометрия,
файловый слой, фронтенд (`frontend/app.js`), воркер уведомлений.

Модель угроз. Сетевого сервера нет — фронтенд общается с бэкендом через мост
`window.pywebview.api.api_request`, который прогоняет запросы к ASGI-приложению
in-process. Ключевое следствие: **любой JavaScript, выполнившийся внутри
WebView, получает полный доступ к бэкенду** (открытие файлов, `subprocess`,
чтение диска). Поэтому XSS в рендеринге заметок эквивалентен выполнению кода.

## Исправлено

### 1. XSS → RCE через обходимый санитайзер Markdown (критично)

`sanitizeRenderedHtml` в `frontend/app.js` вырезал скрипты и обработчики `on*`
регулярными выражениями. Такой подход обходится тривиально, например:

- `<img/src=x/onerror=alert(1)>` — разделителем атрибута в HTML может быть `/`,
  а правило `\son\w+` требовало пробел, поэтому обработчик не вырезался;
- `<img src="x>" onerror=alert(1)>` — символ `>` внутри значения атрибута
  обрывал сопоставление тега, и `onerror` проходил дальше;
- `<a href="java&#09;script:alert(1)">` — схема `javascript:`, разбавленная
  HTML-сущностью/пробелом, не распознавалась.

Любой такой фрагмент в описании карточки (в т.ч. из импортированного или
общего хранилища, из ответа ИИ) выполнил бы код в WebView и через мост —
нативные операции.

Исправление: авторитетный санитайзинг переведён на **инертный DOM**
(`DOMParser`, который не исполняет скрипты и не запускает `onerror`) — он
устойчив к mXSS-обходам. В Web Worker (где DOM недоступен) используется
усиленный строковый фильтр как первый проход, а результат воркера теперь
повторно чистится DOM-санитайзером на главном потоке перед вставкой в
`innerHTML`. Легитимный контент (YouTube-встраивания, `<video>/<audio>`,
вывод KaTeX, ссылки `https/file/data:image`) сохранён.

### 2. Path traversal при открытии/раздаче вложений (средне)

Эндпоинты склеивали пользовательское имя с папкой вложений без проверки выхода
за её пределы:

- `POST /system/open-file` — `../../…` позволял открыть произвольный файл ОС;
- `POST /system/open-link` (ветка внутренних `/doe/`) — то же;
- `GET /doe/{path}` (`serve_attachment` в `main.py`) — чтение произвольного
  файла по `/doe/../../…`.

Исправление: добавлена проверка `_is_within_dir` (resolve + вложенность), как
уже было сделано в `/system/delete-file`. Запросы за пределы папки вложений
получают `403`.

### 3. Ужесточение PowerShell-фолбэка уведомлений (низко)

В `notify_worker.py` (Windows) `task_id` подставлялся в PowerShell-скрипт без
приведения типа. Теперь принудительно `int(task_id)` — исключает любую
инъекцию через это поле. `title`/`message`/`vault_path` уже экранировались.

## Проверено — уязвимостей не найдено

- **Крипто хранилища** (`vault_crypto.py`): scrypt (N=2¹⁵) для KDF, AES-256-GCM
  пофайлово, уникальный nonce на чанк, AAD против перестановки/усечения,
  сравнение хэшей константным временем. Пароль нигде не хранится. Сделано
  добротно.
- **Биометрия** (`biometric.py`): ключ в Keychain под Secure Enclave ACL;
  dev-фолбэк — файл `0600` под гейтом Touch ID.
- **SQL**: «сырые» запросы в поиске (`system.py`) собирают только имена колонок
  и операторы, все значения идут через bind-параметры (`:key`). Инъекции нет.
- **Автоматизации**: выполняют только сортировку/очистку/повтор карточек — shell
  не запускают.
- **Загрузка ИИ-моделей**: URL строится из захардкоженного реестра репозиториев
  HuggingFace по HTTPS; произвольный URL не подставляется.
- **subprocess-вызовы**: везде списочная форма аргументов (без `shell=True`),
  инъекция аргументов невозможна.
- **Загрузка вложений**: имена файлов очищаются `_safe_attachment_name`.

## Второй проход — 2026-07-11 (исправлено)

### 5. Запуск исполняемых файлов одним кликом по ссылке (средне)

`POST /system/open-file`, `POST /system/open-link` и мост
`open_local_path` (wrapper) отдавали путь в `open`/`os.startfile`/`xdg-open`.
Ссылка/вложение из недоверенного хранилища (импорт, общий vault, ответ ИИ)
вида `[x](/path/Evil.app)` или `~/Downloads/run.command` запускала бы код
одним кликом. Добавлена проверка `_is_dangerous_executable`: типы, которые ОС
автозапускает (`.app/.command/.sh/.exe/.bat/.ps1/.vbs/.jar/.desktop/.lnk` и
др.), больше не открываются — обычные документы открываются как прежде.

### 6. Слабая проверка вложенности в `POST /system/delete-file` (низко)

Containment проверялся строковым `startswith`, который обходится
каталогом-соседом с общим префиксом (`…/attachments_evil` начинается с
`…/attachments`). Заменено на `_is_within_dir` (resolve + вложенность).

### 7. Экранирование путей в PowerShell-фолбэке уведомлений (низко)

`notify_worker.py`: `icon_path`/`doe_exe_path` подставлялись в PowerShell без
экранирования — апостроф в пути установки/хранилища разорвал бы строку.
Добавлено `''`-экранирование (как уже было у `title/message/vault_path`).

### 8. `<iframe>` с произвольным src в рендере заметок (низко)

Санитайзер пропускал любой `<iframe>`. Добавлен белый список хостов встраивания
(YouTube/Vimeo `embed`, локальный `file://` для PDF) в обоих санитайзерах
(DOM и строковый fallback); остальные iframe вырезаются.

Углублённый разбор ранее бегло просмотренных модулей: крипта хранилища,

### 9. Stored XSS → RCE в канвасе «Пространство» (критично)

`renderMd` в `frontend/space.js` вставлял вывод `marked.parse(...)` прямо в
`innerHTML` элементов пространства **без санитайзинга** (в отличие от
описаний карточек в `app.js`, где всегда прогоняется `sanitizeRenderedHtml`).
`marked` не чистит HTML, поэтому текст элемента с `<img src=x onerror=…>` или
`<script>` выполнил бы код в WebView, а через мост `window.pywebview.api` —
нативные операции (открытие файлов, subprocess). Вектор реалистичен: документ
пространства хранится в vault (`doe/…`) и мог прийти из импортированного/общего
хранилища. Исправление: вывод `marked` теперь проходит через тот же
DOM-санитайзер `sanitizeRenderedHtml`, что и остальной markdown (функция
глобальна, `app.js` грузится раньше `space.js`). Легитимный markdown сохранён.

### Проверено на этом проходе — уязвимостей не найдено

- **Крипта** (`vault_crypto.py`): scrypt (N=2¹⁵) с раздельными солями для
  проверочного хэша и ключа; AES-256-GCM пофайлово, уникальный nonce на чанк,
  AAD против перестановки/усечения, `hmac.compare_digest`. Путь в v2-контейнере
  проверяется на traversal при расшифровке. Пароль нигде не хранится.
- **Биометрия** (`biometric.py`): ключ в Keychain под Secure Enclave ACL
  (`BiometryCurrentSet`); dev-фолбэк — файл `0600` под гейтом Touch ID.
- **AI-роуты** (`ai.py`): `model_path` всегда собирается из захардкоженного
  реестра `AVAILABLE_MODELS` (произвольный путь не подставить); все действия
  ИИ идут через ORM-слой с bind-параметрами; `_search_board` — `ILIKE` с
  параметром.
- **Автоматизации**: только сортировка/очистка/повтор карточек через ORM —
  ни `eval/exec`, ни shell.
- **`config.spawn_notification_worker`**: `subprocess.Popen` со списком argv
  (без `shell=True`) — инъекция аргументов невозможна.
- **`fs_store.py`**: `yaml.safe_load`/`safe_dump`; имена файлов из заголовков
  проходят `_sanitize` (вырезает `\ / : * ? " < > |`, ведущие точки) — traversal
  через заголовок карточки невозможен.
- **`memory_service`/`ai_service`**: память пишется в `vault/doe/
  user_ai_profile.txt` (путь серверный, не пользовательский).
- **Сетевой сервер отсутствует**: POST на `127.0.0.1:8000` в `notify_worker.py`
  — мёртвый фолбэк (соединение отклоняется → просто запускает приложение).
  Реального слушающего сокета нет, вся связь фронт↔бэк идёт через in-process
  ASGI-мост. Порт наружу не открыт.

## Четвёртый проход — 2026-07-11 (исправлено)

Свежий сквозной разбор. Окно приложения грузится по `file://` (origin
`null`), поэтому любой HTML, попавший в WebView, исполняется с доступом к мосту
`window.pywebview.api` — это ключ к оценке трёх новых находок.

### 10. `<iframe src="file://…">` с произвольным локальным HTML → RCE (критично)

Оба санитайзера (`sanitizeRenderedHtml` и строковый fallback в `app.js`;
`space.js` их переиспользует) пропускали **любой** `file://` в `<iframe>` —
разрешение было задумано только под встроенный PDF. Но вложения кладутся в
папку `doe/`, а имя/содержимое файла ничем не ограничены: заметка из
импортированного/общего хранилища вида
`<iframe src="file:///…/doe/evil.html">` загрузила бы **локальный HTML в
привилегированном WebView** (окно само на `file://`), а тот через мост —
нативные операции (`subprocess`, чтение диска). Исправление: `file://`-iframe
разрешён только если путь оканчивается на `.pdf` (до `#`/`?`):
`^file://[^?#]*\.pdf(?:[?#]|$)` — в обоих санитайзерах. Встраивание PDF, а также
YouTube/Vimeo сохранены; HTML/SVG/произвольные локальные файлы во фрейме
блокируются.

### 11. Инъекция в инлайновый `<script>` через `theme`/`language` (низко/средне)

`serve_index` (`main.py`) **и живой путь загрузки окна** `runtime_index_url`
(`wrapper.py`) подставляли `theme`/`language` из `~/.doe_config.json` прямо в
инлайновый `<script>` (`localStorage.setItem('doe-theme', '{theme}')` и т.п.) без
экранирования. Значение вида `</script>…` означало бы внедрение кода в
привилегированный WebView с сохранением между перезапусками (персистентный
XSS→RCE), если недоверенный ввод когда-либо дойдёт до настроек. Исправление: в
обоих местах `theme`/`lang` жёстко приводятся к белому списку (`light`/`dark`,
`ru`/`en`) перед подстановкой.

### 12. Обход блок-листа исполняемых типов + пробелы в самом списке (средне)

Проверка `_is_dangerous_executable` (`system.py`) и инлайновая в
`open_local_path` (`wrapper.py`) брали суффикс «как есть». ОС (особенно Windows)
отбрасывает завершающие точки/пробелы в имени, поэтому `evil.exe.` и `evil.exe `
давали суффикс `''`/`.exe ` и **обходили проверку**, но `os.startfile` всё
равно запускал `.exe`. Исправление: имя нормализуется (`rstrip(" .\t\r\n")`)
перед извлечением расширения; при ошибке определения типа — отказ в открытии
(fail-safe). Заодно блок-лист дополнен известными авто-запускаемыми форматами:
`.pkg .mpkg .webloc .mobileconfig` (macOS), `.settingcontent-ms .url .scf .xll
.ws .job .chm .hlp .ps1xml .vbscript .msh/.msh1/.msh2(+xml)` (Windows),
`.appimage` (Linux) — в `system.py` и `wrapper.py`.

### Проверено на этом проходе — уязвимостей не найдено

- **SQL-поиск** (`/search`, `_search_by_tags`): в `text()` интерполируются
  только имена колонок и операторы AND/OR из дерева разбора; все значения — через
  bind-параметры (`:tagN`, `:key`). Инъекции нет.
- **`open-link`/`open-file`/`delete-file`/`reveal-folder`**: traversal закрыт
  `_is_within_dir` (resolve+вложенность), исполняемые типы блокируются.
- **Мост `api_request`**: тело base64→ASGI in-process; сетевого сокета нет.
- **CSS через `ui_font`/`custom_font`** (`app.js`): значение уходит в
  `style.setProperty('--font-main', …)` — выйти за пределы значения свойства и
  внедрить правило/скрипт нельзя.

## Остаточные риски (осознанно приняты)

- `POST /system/open-link` по-прежнему открывает обычные документы/папки по
  абсолютному пути (это фича Obsidian-стиля). Опасные исполняемые типы теперь
  блокируются (п. 5), но подтверждение перед открытием документа из
  недоверенной ссылки можно добавить как доп. рубеж.
- `Content-Security-Policy` **добавлен** (слоёный, defense-in-depth) — `<meta>`
  в начале `<head>` `frontend/index.html` (окно грузится по file://, поэтому CSP
  задаётся мета-тегом; он попадает и в генерируемый `runtime_index_url`).
  Жёстко закрыты вторичные векторы: `object-src 'none'`, `base-uri 'none'`
  (перехват относительных URL через `<base>`), `form-action 'none'`, а `frame-src`
  повторяет whitelist санитайзера (YouTube/Vimeo + `file:` для PDF). В `script-src`
  сохранён `'unsafe-inline'`: весь рабочий JS вынесен в отдельные файлы, но в
  разметке остаются инлайновые обработчики (`onclick` и др.). Полный запрет
  `'unsafe-inline'` (снятие ~42 инлайн-обработчиков на `addEventListener`) даст
  защиту от обхода санитайзера и остаётся следующим шагом ужесточения.
- Загруженные ИИ-модели не проверяются по хэшу (полагается на HTTPS из
  захардкоженного реестра HuggingFace).
